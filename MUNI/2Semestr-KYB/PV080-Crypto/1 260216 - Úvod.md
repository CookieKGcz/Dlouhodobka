## Cryptographic basics
### Terminology
- Cryptography - art and science of securing data against adversaries
	- Public broadcast - encryption is necessity
- Steganography - art of concealing data within other data
	- when encryption is illegal/suspicious
- Cryptanalysis - art and science of revealing weaknesses in cryptog.
- Cryptology = Cryptography + Cryptoanalysis
### Symmetric encryption
#### Encryption
- specific recipient
	- msg = 'My pin is 1234'
- Transformation must involve secret known to recipient - encryption

#### Encryption, Encoding as mappings
- One to one - allows for ==reverse== mappings (decoding, decryption)
- Encoding: different domains (typically) - $E: X \to Y$
	- characters to integers/bytes/bits
- Encryption: same domain - $E_{K}: X \to X$
	- one-to-one + same domain $X \implies E_{K}$ is permutation
	- X is set of blocks of bits or integers
	- ![[Pasted image 20260216104945.png | 500]]
#### Encryption and Decryption - terminology
- Encryption transforms data (plaintext) into a form that is unintelligible (ciphertext) without a possession of critical parameter (key)
	- 2 inputs: plaintext ($PT$), encryption key ($K_{E}, k$)
- Decryption facilitates the recovery of plaintext with the use of decryption key
	- 2 inputs: ciphertext ($CT$), decryption key ($K_{D}, k'$)
- Encryption provides data confidentiality = access to data semantics
#### Encryption, Encoding - key size and usage
- Mappings $E, E_{K}$ - applied to all data blocks (block of bits, characters)
	- Encoding - fixed $E$ is applied
	- Encryption - K (and $E_{K}$) is fixed or is changing
- Attacks targeting key:
	1. Brute force - try all possible keys
	2. Statistical attacks - correlation between PT, CT, and K - smaller key set to test
- Key
	- Small key - changing - brute force attack feasible if fixed
	- Large key - fixed but must be large enough

#### Attacks and countermeasures
- Attacks targeting key:
	1. Try possible keys: decrypt and evaluate obtained candidate plaintext (PT).
	2. Compute appropriate statistic in ciphertext (CT) and select (sort) candidates for K.
- Basic attacks in classical cipher:
	1. For K in 0-26: decrypt and check meaningfulness of PT.
	2. Compute letter frequencies in CT - select keys for EN frequencies.
- Countermeasures:
	1. Testing all keys should be infeasible - key larger then 100 bits.
	2. Cipher design that produce random (looking) CT.
#### Key size and usage in practice
- Small key (1 bit) - _specific_ key bit for each data bit
	- Bits separately processed
	- Two simple mappings (table)
	- Simple relation (PT, CT) <-> key - unique key !
- Large key (>= 128 bits) - fixed key for each data block
	- Blocks (of 64 up to 4096 bits) separately processed
	- Complex algorithm - mixes K with data
	- Complex relation (PT, CT) <-> key - same size

#### Vernam cipher/OTP
- Symmetric alg., stream cipher – encrypts plaintext by one bit at a time.
- ![[Pasted image 20260216123536.png | 400]]
- If the keystream is a sequence of truly random, independent bits that is never re-used, then this is an unbreakable ane-time pad. Practical encryption systems aim to mimic the one-time pad using shortcuts without compromising security.

- Perfect (secure) cipher => ==key as long as the plaintext== => impractical
- Also called One time pad - key for one(!) use only => expensive/impractical
#### Reused keystream issue (Vernam cipher)
- Reused keystream can be canceled out (XOR is commutative).
- ![[Pasted image 20260216123909.png | 400]]
- **XOR properties:**
	- $A \oplus B = B \oplus A$
	- $A \oplus A = 0$
	- $A \oplus B = C \implies A = C \oplus B$

	- $CT_{1} \oplus CT_{2}$
	- $= PT_{1} \oplus PT_{2} \oplus (KS \oplus KS)$
	- $= PT_{1} \oplus PT_{2}$
- Keystream must be unique – can be canceled out or extracted from PT, CT
- Similar problem for redundant keystream (repeated parts): abc abc abc …
#### Stream ciphers
- Designed to be deterministic RNG (stateful):
	- generate large random sequence – keystream,
	- initialized by pair 𝐾,𝐼𝑉 (initialization vector) – ==pair== must be unique
- Stream ciphers characteristic:
	- Two types: Synchronous ($𝐶𝑇 = 𝑃𝑇 \oplus 𝑘𝑒𝑦𝑠𝑡𝑟𝑒𝑎𝑚$), asynchronous.
	- Extremely fast bitwise ops only (XOR, AND, OR, shift).
- Common stream ciphers - ChaCha20
- RC4 is weak
#### Initialization vector (IV)
- Additional parameter:
	- randomizes CT for some K, PT.
- Does not need to be random, but must be unique!
- Same (IV and K) => same keystream
- ![[Pasted image 20260216124846.png | 300]]
#### Stream vs block
- Stream vs. block ciphers:
	- Encryption/decryption
		- bit/byte processed at the time vs larger block (64b, 128b, 256b)
	- Speed - slower vs. faster
- Stream ciphers:
	- Widely used for hardware implementation.
	- Efficient for unknown file size, continuous stream (audio, video)
- Block ciphers:
	- suitable for software implementation,
	- efficient for known file size – parallel processing possible.
	- padding may be needed (but can lead to attacks).

#### Block ciphers
- Permutations operating on blocks of fixed size (typically 128b, 256b).
- Padding: incomplete last block must be completed (padded):
	- several padding schemes (e.g., zero padding),
		- to remove padding one needs to distinguish between padding and data!
			- Length of padding (written) in the last byte is typically used;
				- msg is always padded!
- Each block cipher have many versions (modes):
	- some use IV (TBD later) – must be unique.
#### DES - Data Encryption Standard
- Data Encryption Algorithm (DEA)
- Still the most widely used encryption scheme (cheap hardware)
- IBM cipher LUCIPHER (H. Feistel, project for Lloyd’s Bank (UK)), modified(!)
	- 128bit key-length reduced to 56 bits (other half for checksum)
- Weak keys (4): $E_{K} (x) = x$
- Semi-weak keys (6 pairs): $E_{K_{1}}(E_{K_{2}}(x))=x$
#### Breaking DES
- 2^56 keys possible – brute force (exhaustive search)
- in 1977 Diffie and Hellman - design (20$)
#### 3DES
- In the second step, the result of the first step, C1 , is decrypted using the second key, K2
	- Since K2 ≠ K1 , this does not result in the original plaintext message.
- In the final step, the result of the second step, C2 , is encrypted using the third key, K3
	- The output ciphertext C3 is the final encrypted message.
	- Recall that ==K3 = K1== in this case, so even though there are three 56-bit keys, the effective key length is only 112-bits.

#### Attacks and attackers I
- Exhaustive search = brute force = try all possible options
- Ciphertext-only = adversary tries to recover plaintext (or the key), given access to ciphertext alone
- Known-plaintext = adversary tries to recover unknown plaintext (or the key) from further ciphertext, given access to some ciphertext and its corresponding plaintext
- Chosen-plaintext = adversary can choose some amount of plaintext and see the resulting ciphertext. Such additional control may allow advanced analysis that defeats weaker algorithms
- Chosen-ciphertext = for a fixed key, adversary can provide ciphertext of her choice, and receive back the corresponding plaintext; the game is to again deduce the secret key, or other information sufficient to decrypt new ciphertext
#### Attacks and attackers II
- Attacker model – clear specification what the attacker CAN do – and even better also what he CANNOT do

- Passive adversary = observes and records, but does not alter information (e.g., ciphertext-only, known-plaintext attacks).
- Active adversary = interacts with ongoing transmissions, by injecting data or altering them, or starts new interactions with legitimate parties (e.g., chosen-plaintext, chosen-ciphertext attacks).

#### Cryptographic algorithm & (vs?) key
- Kerckhoffs’ principle – a system’s security should not rely upon the secrecy of its design details!
	- the system must not require secrecy and must be able to be stolen by the enemy without causing trouble
	- C. Shannon: the enemy knows the system
	- If you hardcode the secret(s) into all system installations (hardware!) then you have to replace all these installations once the adversary learns the secret(s)
- Algorithms
	- with keys: ciphers (asymmetric, symmetric), MAC, etc.;
	- without keys (unkeyed) – hash functions, RNG.
#### Randomness
- Crypto relies on hard to guess and unique values:
	- keys, 𝐼𝑉s, nonces, challenges, etc
- Random values (large pool) are simultaneously ⇒ hard to guess, unpredictable, unique

- RNGs:
	- Truly random (non-deterministic) number/sequence:
		- physical process (radio decay, thermal noise, etc.) – aperiodic, slow.
	- Pseudorandom (deterministic) number/sequence:
		- software function – periodic, fast,
		- initialized (internal state) by seed – fully determines next random data.
#### TRNG or PRNG?
- How to generate key for stream or block cipher?
- PRNG – fast but deterministic;
	- compromised internal state (or seed) => predictable next (random) value(s)
	- ![[Pasted image 20260216132731.png | 150]]
- TRNG – slow but non-deterministic;
	- unpredictable!
	- ![[Pasted image 20260216132712.png | 150]]

- ==System RNGs (e.g., dev/urandom) combine both==:
	- PRNG seeded by TRNG – fast (PRNG) with non-determinism (TRNG)
#### Sources of randomness
- Recommended PRNGs (CSPRNG – cryptographically secure PRNG):
	- Based on crypto-primitives:
		- ChaCha20 (stream), AES CTR mode(block), iterative hashing (hash, hash of hash, …).
- Recommended TRNG – special designs:
	- Yarrow, Fortuna (dev/random, dev/urandom files)
- Not recommended:
	- ==Leak state with generated random values – predictable!==
	- ==Standard random functions – rand, Java.util.Random.==
	- ==Mersenne twister, Dual_EC_DRBG, RC4==
